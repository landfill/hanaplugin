# 태깅 정의서 스킬

화면의 특정 영역(버튼, 배너 등) 클릭 시 **Amplitude에 전달할 항목**을 정의하는 업무를 가이드하고, 확정된 내용을 **마크다운 데이터 문서**로 정리한 뒤 스크립트로 **태깅 정의서(PPT 또는 엑셀)**를 생성한다.

## 필수 도구 인터페이스

<MANDATORY>
**[MANDATORY RULE — TOOL INTERFACE]**
모든 단계의 사용자 질문은 **반드시 AskQuestion 도구**를 사용한다. 일반 채팅 텍스트로 질문하지 않는다.
각 질문은 **구조화된 선택지(options)**를 포함해야 하며, 자유입력이 필요한 경우에도
"직접 입력" 옵션을 선택지 중 하나로 제공한다.
이 규칙은 1단계~4단계 전 구간에 적용된다. 예외 없음.
</MANDATORY>

**AskQuestion 호출 규칙**:
- 질문마다 고유 `id`를 부여한다 (예: `"step1-service-name"`, `"step2-event-action"`)
- 선택지는 최소 2개, 자유입력이 필요한 항목은 마지막에 `"직접 입력"` 옵션을 포함한다
- 한 번에 여러 질문을 묶어도 된다 (AskQuestion은 `questions` 배열을 지원)
- `allow_multiple: true`는 복수 선택이 필요한 경우에만 사용한다

## 맥락

- **대상 서비스 예시**: www.hanatour.com, m.hanatour.com (스킬 서두 맥락으로 명시 가능).
- **태깅 범위**: 서비스 전역에서 이미 수집 중인 항목은 "기본 태깅"으로 두고, **추가로 정의할 대상**(버튼 클릭, 신규 화면 오픈, 부가 기능 등)만 이 스킬로 정의한다.
- **전역 공통 7개 제외**: 아래 항목은 정의서/스킬 출력에서 **제외**한다. 해당 클릭에만 특화된 property만 정의한다.
  - `bSessionId`, `isLogin`, `locationPathname`, `prePage`, `resPathCd`, `queryString`, `domain`
- **Property 네이밍**: 신규 property 제안 시 **camelCase**를 기본으로 적용한다.
- **이벤트 영문명**: **snake_case**로만 통일한다. camelCase·혼용 금지. 가능하면 **짧게** 제안한다(예: `click_cert_btn`, `cert_popup_close`, `cert_popup_print`). 프리픽스(`view_`, `click_`, `load_` 등) 사용 권장. 규칙 없을 때는 `[액션]_[대상]_[상세]` 또는 `[대상]_[액션]` 형태.
- **속성명 예시**: `areaCd`, `saleProdCd`, `depDay`, `saleProdNm` 등 camelCase 유지.
- **문체**: PPT/엑셀 내부 문구는 **한국어**, 말맺음은 **명사형** (예: 적용한다 → 적용).

## 사용자가 제공할 것

- **필수**: 어떤 화면인지, 어떤 기능/버튼/액션을 추적할지 **텍스트 설명**. 정의할 이벤트 목록(호출 시점 + 한글 이름), 이벤트당 수집 속성(속성명 영문, 한글 설명, property 구분, value, description).
- **권장**: Event Name 영문, Event Category, 기획서·화면정의서 해당 부분 **텍스트 붙여넣기** 또는 **슬라이드 이미지 첨부**.
- **참고**: 기존 태깅 범위 경계("추가로 정의할 대상"만 요청 시). 기존 태깅 정의서(엑셀/PPT) 샘플이 있으면 제공 시 스킬 문구 구체화에 활용.

## 워크플로우

```
1단계: 맥락 수집 → [사용자 응답·확인] →
2단계: 이벤트 수집 → [사용자 응답·확인] →
3단계: 프로퍼티 수집(이벤트별 인터뷰) → [사용자 응답·확인] →
4단계: 시각 자료 → [사용자 응답·확인]
    ↓
정리·사용자 컨펌 → 확정
    ↓
데이터를 마크다운 문서(데이터 .md)로 저장
    ↓
Python 설치 여부 확인 → 출력 형식(PPT/엑셀) 선택 또는 폴백(마크다운만)
    ↓
run_generate.py 실행 → .pptx/.xlsx 생성 (또는 마크다운만 채팅 출력)
```

**단계 분리**: 한 턴에서 한 단계만 진행. 다음 단계는 이전 단계가 끝나 사용자 응답을 받은 뒤 다음 메시지에서 시작.

<HARD-GATE>
**[MANDATORY RULE — NO PREMATURE GENERATION]** 태깅 항목이 모두 정의되고 사용자 **컨펌**이 끝나기 전에는 파일(.pptx/.xlsx) 생성을 시작하지 않는다. 확정된 내용만 최종 문서에 반영한다.
</HARD-GATE>

<HARD-GATE>
**[MANDATORY RULE — ONE STEP AT A TIME]** 한 번에 한 단계만 진행한다. 1단계 질문·수집 후 사용자 응답을 받고, 1단계가 끝났음을 확인한 **뒤에만** 2단계를 시작한다. 2단계 완료 후에만 3단계, 3단계 완료 후에만 4단계. **1단계와 2단계(또는 다른 단계들)를 같은 턴에서 동시에 묻지 않는다.**
</HARD-GATE>

## 1단계: 맥락

**이 단계만** 진행한다. 1단계 수집이 끝나 사용자 응답을 받기 전에는 2단계(이벤트) 질문을 하지 않는다.

**반드시 AskQuestion 도구**로 수집한다. 아래 패턴을 따른다:

```
AskQuestion 호출 패턴 (1단계):
title: "1단계: 맥락 수집"
questions:
  - id: "step1-service-name"
    prompt: "서비스명을 알려 주세요 (파일명·표지에 사용됩니다)"
    options: ["하나투어 닷컴", "하나투어 모바일", "직접 입력"]
  - id: "step1-team"
    prompt: "팀/소속을 알려 주세요"
    options: ["기획팀", "데이터팀", "직접 입력"]
  - id: "step1-author"
    prompt: "작성자 이름을 알려 주세요"
    options: ["하니", "직접 입력"]
  - id: "step1-screens"
    prompt: "태깅 대상 화면 목록이 있나요? (있으면 개요 페이지에 반영)"
    options: ["있음 (다음에 입력)", "없음 (개요 페이지 제거)"]
```

- 서비스명(파일명·표지용)
- 팀/소속, 작성자(기본 "하니"), 작성일
- "태깅 대상 화면 목록이 있나요?" → 있으면 3페이지(개요) 반영, 없으면 개요 페이지 제거

1단계 수집이 끝나면 AskQuestion으로 확인한 뒤, **그다음 메시지에서** 2단계로 넘어간다.

## 2단계: 이벤트 수집

**1단계(맥락)가 완료·확인된 후에만** 이 단계를 시작한다. 2단계만 진행하며, 3단계(프로퍼티) 질문은 하지 않는다.

**지금은 "어떤 사용자 행동(이벤트)을 찍을지"만 수집하는 단계입니다. 프로퍼티(속성)는 다음 단계에서 묻습니다.**

질의 항목·순서·문구는 [reference.md의 질의 표준지침(2단계)](reference.md#2단계-이벤트-수집--고정-질의-항목)을 **반드시** 따른다. 호출마다 문의 항목이 달라지지 않도록 아래 4항목을 **매 이벤트마다 동일한 순서**로 묻는다.

1. **어떤 행동인지**: "어떤 사용자 행동을 찍을까요? (예: 버튼 클릭, 탭 선택, 팝업 닫기)"
2. **호출 시점**: "이 이벤트는 **언제** 호출되나요? (예: OO 버튼 클릭 시, 탭 선택 시)"
3. **이벤트 영문명**: [reference.md 이벤트 영문명 프리셋](reference.md#이벤트-영문명-프리셋)을 참고해 **축약형·동일 용어**로 제안. 예: `click_*_btn`, `*_popup_close`, `search_result`, `*_tab`. "영문 이벤트명을 쓸게요. [제안]. 이대로 쓸까요?"
4. **이벤트 한글명·신규/기존**: "한글 이름은 '[한글명]', 구분은 [신규/기존]으로 할까요?"

**반드시 AskQuestion 도구**로 질문한다. 아래 패턴을 따른다:

```
AskQuestion 호출 패턴 (2단계 — 이벤트 1개당):
title: "2단계: 이벤트 수집"
questions:
  - id: "step2-action-type"
    prompt: "어떤 사용자 행동을 찍을까요?"
    options: ["버튼 클릭", "탭 선택", "팝업 닫기", "팝업 열기", "페이지 로드", "검색", "스크롤", "직접 입력"]
  - id: "step2-trigger"
    prompt: "이 이벤트는 언제 호출되나요?"
    options: ["OO 버튼 클릭 시", "탭 선택 시", "페이지 진입 시", "팝업 닫기 시", "직접 입력"]
  - id: "step2-event-name-en"
    prompt: "영문 이벤트명을 제안합니다: [제안]. 이대로 쓸까요?"
    options: ["네, 그대로 사용", "직접 입력"]
  - id: "step2-event-name-kr"
    prompt: "한글 이름은 '[한글명]', 구분은 [신규/기존]으로 할까요?"
    options: ["네, 그대로 사용", "신규로 변경", "기존으로 변경", "직접 입력"]

이벤트 1개 완료 후:
  - id: "step2-more-events"
    prompt: "같은 화면에서 더 정의할 이벤트가 있나요?"
    options: ["있음 (계속 추가)", "없음 (2단계 종료)"]
```

- "같은 화면에서 더 정의할 **이벤트**가 있나요?" → 있으면 위 4항목 반복. 없으면 2단계 종료.

2단계(이벤트 목록)가 끝나면 AskQuestion으로 확인한 뒤, **그다음 메시지에서** 3단계로 넘어간다.

## 3단계: 프로퍼티 수집 (인터뷰 기반)

**2단계(이벤트)가 완료·확인된 후에만** 이 단계를 시작한다. 3단계만 진행하며, 4단계(시각 자료) 질문은 하지 않는다.

**지금은 "이 이벤트에서 어떤 속성(프로퍼티)을 함께 보낼지"만 수집하는 단계입니다. 이벤트 추가/변경은 묻지 않습니다.** 필요 시 "지금은 이벤트가 아니라, 이 이벤트에 붙일 속성(프로퍼티)을 정하는 단계예요."라고 한 줄 안내한다.

질의 항목·순서는 [reference.md의 질의 표준지침(3단계)](reference.md#3단계-프로퍼티-수집--고정-질의-항목)을 따른다. 프로퍼티는 **한 번에 자동으로 정의하지 않는다**. 각 이벤트마다 **질의응답**으로 아래 순서를 유지한다.

**진행 순서(이벤트별, 고정)**:

1. **측정 목적(KPI)**: "이 **[이벤트 한글명]**을 찍어서 **무엇을 알고 싶나요?** 어떤 숫자나 비율을 보려는지 한 줄로 알려 주세요."
   - 예: "보험가입증서 버튼을 새로 냈을 때, 얼마나 클릭하는지, 증서 팝업에서 그냥 닫는지 인쇄하는지 비율을 보고 싶다" → 클릭 수, 팝업 내 액션(닫기/인쇄) 구분이 필요.
2. **필요 프로퍼티 도출**: "(그 목적에 맞춰) 이 이벤트에 넣을 **속성(프로퍼티)**은 [제안 목록]입니다. 더 넣거나 빼고 싶은 게 있나요?"
   - CTA의 모든 레이블을 기본으로 넣지 않음. **의미 있는 분석 단위**만.
3. **value(샘플) 확인**: "프로퍼티 **[속성명]**의 샘플 값(value)은 무엇으로 할까요? (예: …) 없으면 제가 예시로 채울게요." → 사용자 입력 있으면 반영, 없으면 자동 생성.
4. **정리**: "한글명·영문명(camelCase)·value·설명·구분(신규/기존/변경) 이렇게 정리했어요. 수정할까요?" 전역 7개는 제외 안내만.

**반드시 AskQuestion 도구**로 질문한다. 아래 패턴을 따른다:

```
AskQuestion 호출 패턴 (3단계 — 이벤트별):
title: "3단계: [이벤트 한글명] 프로퍼티 수집"
questions:
  - id: "step3-kpi-[이벤트번호]"
    prompt: "이 [이벤트 한글명]을 찍어서 무엇을 알고 싶나요? (클릭 수, 전환율 등)"
    options: ["클릭 수/비율 측정", "전환율 측정", "사용 패턴 분석", "A/B 비교", "직접 입력"]
  - id: "step3-props-[이벤트번호]"
    prompt: "(목적에 맞춰) 속성은 [제안 목록]입니다. 더 넣거나 빼고 싶은 게 있나요?"
    options: ["네, 그대로 사용", "추가할 속성 있음", "제거할 속성 있음", "직접 입력"]
    allow_multiple: true
  - id: "step3-value-[이벤트번호]"
    prompt: "프로퍼티 [속성명]의 샘플 값(value)은?"
    options: ["에이전트가 예시로 채움", "직접 입력"]
  - id: "step3-confirm-[이벤트번호]"
    prompt: "정리 결과를 확인해 주세요. 수정할까요?"
    options: ["확인 완료", "수정 필요"]
```

**주의**: 화면에 있는 CTA/버튼 레이블을 **일괄로 프로퍼티**로 넣지 않는다. "이 이벤트로 답하려는 질문(KPI)"에 필요한 것만 추가한다.

3단계(프로퍼티)가 끝나면 AskQuestion으로 확인한 뒤, **그다음 메시지에서** 4단계로 넘어간다.

## 4단계: 시각 자료

**3단계(프로퍼티)가 완료·확인된 후에만** 이 단계를 시작한다. 4단계만 진행한다.

**반드시 AskQuestion 도구**로 질문한다:

```
AskQuestion 호출 패턴 (4단계):
title: "4단계: 시각 자료"
questions:
  - id: "step4-has-image"
    prompt: "해당 화면/영역 캡처 이미지가 있나요?"
    options: ["있음 — 채팅에 붙여넣기", "있음 — 파일 경로 제공", "없음 (와이어프레임만)"]
```

- "해당 화면/영역 캡처 이미지가 있나요?" → 있으면 4페이지 왼쪽 40% 사용, 없으면 와이어프레임 영역만 확보

**제공 방식**(둘 중 편한 방식):

| 방식 | 설명 |
|------|------|
| **채팅에 붙여넣기** | 캡처한 이미지를 **채팅창에 직접 붙여넣기**(드래그 앤 드롭 또는 Ctrl+V / Cmd+V). 에이전트가 대화 맥락에서 이미지를 확인하고, 필요 시 프로젝트에 저장한 뒤 PPT 생성 시 경로로 넘길 수 있음. |
| **저장된 파일 경로** | 이미 **물리적으로 저장된 이미지 파일**이 있으면 파일 경로를 텍스트로 알려 줌(예: `docs/screens/서브메인_탭.png`). 에이전트가 해당 경로의 파일을 읽어 출력물에 반영할 수 있음. |

- 이미지가 없으면 와이어프레임 영역만 확보하고 진행.
- **어떤 이미지가 어떤 태깅(이벤트)에 해당하는지** 구분하는 방법:
  - **순서로 매핑**: 이벤트 정의 **순서와 동일하게** 이미지를 제공(첫 번째 이미지 → 첫 번째 이벤트 슬라이드, 두 번째 → 두 번째 …). 여러 장 제공 시 "첫 번째는 [이벤트명], 두 번째는 [이벤트명]"처럼 한 줄로 명시해 주면 더 명확함.
  - **이벤트별 경로**: 데이터 마크다운(`태깅정의_데이터.md`)에 **이벤트 블록마다** `- 캡처 경로: 상대경로/파일명.png` 를 넣으면, PPT 생성 스크립트가 해당 이벤트 슬라이드 **왼쪽 40%에만** 그 이미지를 배치하고 오른쪽 60%에 태깅 표를 둠. 채팅에 붙여넣기한 이미지는 에이전트가 먼저 저장한 뒤, 저장 경로를 데이터 .md의 각 이벤트에 반영하면 됨.
- PPT 생성 스크립트는 이벤트 순서대로 슬라이드를 만들고, 각 슬라이드에 해당 이벤트의 `캡처 경로`가 있으면 왼쪽에 이미지, 오른쪽에 표를 배치함.

## 데이터 형식: 마크다운만 사용

JSON은 사용하지 않는다. 수집·컨펌한 결과는 **한 번만** 구조화하여 **마크다운 문서**로 정리한다.

- **단일 데이터 소스**: 이벤트 목록, 이벤트별 property·value·description·property 구분을 **마크다운 제목·목록·표**로만 표현.
- **폴백(Python 미설치 시)**: 태깅 정의 항목을 **마크다운 표**만 채팅으로 출력. 사용자는 이를 복사해 엑셀에 붙여넣거나, 나중에 Python 설치 후 같은 데이터로 파일 생성 요청 가능.
- **파일 생성 시**: 동일한 내용을 [데이터 마크다운 형식](reference.md#데이터-마크다운-형식)에 맞춰 `태깅정의_데이터.md` 등으로 저장한 뒤, 아래 스크립트에 해당 파일 경로를 넘겨 실행한다.

## PPT 템플릿 (스킬 포함)

스킬 폴더에 PPT 템플릿이 포함되어 있다. 스크립트는 이 템플릿을 채워 .pptx를 생성한다.

- **경로**: `template/태깅정의서_템플릿.pptx` (스킬 루트 기준, 즉 `skills/tagging-definition/template/태깅정의서_템플릿.pptx`).
- **템플릿 반영 원칙**: 템플릿의 **폰트·서식 스타일만** 유지한다. **문구와 캡처 이미지는 반드시 우리 데이터/이미지로 대체**하며, 템플릿 문구가 그대로 남아 중복되거나 겹치지 않도록 스크립트가 기존 내용을 비운 뒤 채운다.
- 프로젝트에 `docs/[서비스명] 태깅 정의서.pptx` 가 있으면 동일한 4페이지 구성의 빈 템플릿으로 간주하며, 스킬용 복사본은 위 경로를 사용한다.

## Python 스크립트로 PPT/엑셀 생성

스킬 폴더의 `scripts/` 에 다음이 있다. **데이터는 마크다운 문서 하나**로만 전달한다.

- **의존성(python-pptx, openpyxl)**: 사용자가 미리 설치하지 않는다. **스크립트 실행 시(런타임)** 에 필요하면 자동 설치한다.
- **진입 스크립트**: `run_generate.py` — 의존성 확인·설치 후 실제 생성 스크립트를 실행한다. 파일 생성 시 **반드시 이 스크립트를 사용**한다.

| 하위 스크립트 | 역할 |
|---------------|------|
| `generate_pptx.py` | 마크다운 데이터 + 템플릿으로 .pptx 생성 |
| `generate_xlsx.py` | 마크다운 데이터로 .xlsx 생성 (PPT와 동일 표 기준) |

**실행 방법** (프로젝트 루트 기준, **런타임에 의존성 자동 설치**):

```bash
# PPT 생성 — run_generate.py가 의존성 확인 후 generate_pptx 실행
python3 skills/tagging-definition/scripts/run_generate.py pptx 태깅정의_데이터.md 출력.pptx

# 엑셀 생성
python3 skills/tagging-definition/scripts/run_generate.py xlsx 태깅정의_데이터.md 출력.xlsx
```

- `run_generate.py` 를 사용하면 `pip install` 은 스킬이 **실행될 때** 필요 시 한 번만 수행된다. 사용자에게 "미리 pip install 하라"고 안내하지 않는다.
- 스크립트는 데이터 마크다운을 파싱하여 .pptx/.xlsx를 만들며, Windows/Mac 둘 다 지원한다.

## 컨펌 후 출력 분기

태깅 항목 정의(1~4단계) 완료 후 **사용자 컨펌**으로 확정한 뒤:

1. **데이터 저장**: 확정된 내용을 [데이터 마크다운 형식](reference.md#데이터-마크다운-형식)에 맞춰 `태깅정의_데이터.md` 로 저장.
2. **Python 설치 여부 확인**: 터미널에서 `python3 --version` 또는 `python --version` 실행, exit code 확인.
3. **Python 설치됨** → 출력 형식(PPT/엑셀) 선택 → **`run_generate.py` 실행**(의존성은 이때 런타임에 자동 설치)으로 .pptx 또는 .xlsx 생성 후 스킬 종료.
4. **Python 미설치** → 사용자에게 선택 안내:
   - **A) 정의 내용만 받고 종료**: Python 설치 안내(예: [python.org](https://www.python.org/downloads/), Windows `winget install Python.Python.3.12`, Mac `brew install python3`) + 태깅 정의 항목만 **마크다운 표**로 채팅 출력 후 종료. 파일 생성 없음.
   - **B) Python 설치 후 파일까지 생성**: 설치 안내 후 사용자가 설치한 뒤 같은 데이터로 다시 실행하도록 안내.
   - **(선택) B') Python 자동 설치 시도**: 시스템 설치 명령 또는 포터블 Python 다운로드 시도. 성공 시 출력 형식 선택 후 .pptx/.xlsx 생성. 실패 시 A와 동일하게 마크다운만 제공.

## 최종 아웃풋 형태

### PPT 4페이지 구성

| 슬라이드 | 제목/역할 | 내용 |
|----------|-----------|------|
| 1 | 표지 | `[서비스명] 태깅 정의서` / YYYY.MM.DD / 팀명 / 작성자 |
| 2 | 이력 | 버전 0.0.1, 변경일=실행일, 변경내용=`[서비스명] 태깅 정의 초안`, 작성자 |
| 3 | 개요 | 태깅 범위(대상 화면 목록). 입력 없으면 페이지 제거 |
| 4 | 태깅 상세 | 호출 이벤트명, 왼쪽 40% 캡처/와이어프레임, 오른쪽 60% 표 |

**4페이지 표 컬럼**(헤더 고정, 행은 입력으로 생성):

| Event property | property 구분 | value | description |
|----------------|---------------|-------|-------------|
| 속성명(영문, camelCase) | 신규/기존/변경 | 샘플 데이터 | 샘플 데이터 설명 |

### 엑셀

PPT와 **동일한 데이터·표 기준**으로 .xlsx 생성. 시트 또는 영역: 표지 → 이력 → 개요 → 이벤트별 태깅 상세(호출 이벤트명 + 표).

## 테스트 방법

스킬의 **스크립트·데이터 형식**은 아래처럼 검증한다. **의존성은 실행 시 자동 설치**되므로 미리 `pip install` 할 필요 없다.

1. **샘플 데이터로 엑셀 생성** (프로젝트 루트에서):
   ```bash
   python3 skills/tagging-definition/scripts/run_generate.py xlsx \
     skills/tagging-definition/scripts/태깅정의_데이터_샘플.md \
     /tmp/태깅정의_테스트.xlsx
   ```
   → 의존성 없으면 설치 후 `저장: /tmp/태깅정의_테스트.xlsx` 출력. 파일에서 표지·이력·개요·이벤트별 시트 확인.

2. **샘플 데이터로 PPT 생성**:
   ```bash
   python3 skills/tagging-definition/scripts/run_generate.py pptx \
     skills/tagging-definition/scripts/태깅정의_데이터_샘플.md \
     /tmp/태깅정의_테스트.pptx
   ```
   → 슬라이드 1~5(표지·이력·개요·이벤트1·이벤트2) 확인. 샘플의 `캡처 경로`는 없는 파일이므로 이미지 없이 표만 배치됨.

3. **전체 스킬 플로우**(1~4단계 질문 → 컨펌 → 데이터 .md 저장 → `run_generate.py` 실행)는 Cursor에서 스킬을 호출한 뒤 대화로 진행해 보면 된다.
