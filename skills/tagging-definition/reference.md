# 태깅 정의서 스킬 — 참고 자료

## 질의 표준지침 (필수 준수)

스킬 사용자는 태깅 이해도가 낮을 수 있으므로 **질의 항목과 문구를 스킬 호출마다 동일하게** 유지한다. 문구만 바꾸지 말고 **수집 항목 자체를 누락·추가·변경하지 않는다.**

<MANDATORY>
**[MANDATORY RULE — TOOL INTERFACE]**
모든 질의는 SKILL.md에서 세션 시작 시 결정된 도구(AskQuestion → AskUserQuestion → 마크다운 텍스트 순)로 수행한다.
각 질문에 **구조화된 선택지(options)**를 포함하고, 자유입력이 필요하면 `"직접 입력"` 옵션을 마지막에 둔다.
</MANDATORY>

<HARD-GATE>
**[MANDATORY RULE — NAMING CONVENTION]**

사용자가 이벤트명 또는 프로퍼티명을 직접 입력한 경우, **반드시 형식을 검증한 뒤 다음 질문으로 넘어간다.** 형식이 맞지 않으면 거부하고 재입력을 요청한다. 사용자 확인 없이 자동 변환하지 않는다.

| 대상 | 규칙 | 위반 예시 | 올바른 예시 |
|------|------|-----------|-------------|
| **이벤트 영문명** | `snake_case` 소문자만. 공백·대문자·camelCase 금지 | `ClickBtn`, `click-btn`, `Click_Btn` | `click_cert_btn` |
| **프로퍼티 영문명(Event property)** | `camelCase`. 첫 글자 소문자, 이후 단어 첫 글자 대문자. 언더스코어 금지 | `fit_tab`, `FitTab`, `fittab` | `fitTab` |

**검증 실패 시 처리**:
1. 위반된 부분을 구체적으로 알린다. 예: "이벤트명은 snake_case여야 합니다. `ClickBtn` → `click_btn` 형태로 수정해 주세요."
2. 올바른 형식의 예시를 제안한다.
3. 사용자가 수정한 값을 다시 검증한 후 진행한다.
</HARD-GATE>

### 2단계: 이벤트 수집 — 고정 질의 항목

**이 단계에서는 "어떤 사용자 행동(이벤트)을 찍을지"만 수집한다. 프로퍼티(속성)는 묻지 않는다.**

아래 **4가지 항목을 매 이벤트마다 동일한 순서로** 묻는다. 생략·통합하지 않는다.

| 순서 | 수집 항목 | 질문/확인 문구 예시 |
|------|-----------|----------------------|
| 1 | **어떤 행동인지** | "어떤 사용자 행동을 찍을까요? (예: 버튼 클릭, 탭 선택, 팝업 닫기)" |
| 2 | **호출 시점** | "이 이벤트는 **언제** 호출되나요? (예: OO 버튼 클릭 시, 탭 선택 시)" |
| 3 | **이벤트 영문명** | "영문 이벤트명을 쓸게요. [프리셋 적용한 제안]. 이대로 쓸까요?" (→ [이벤트 영문명 프리셋](#이벤트-영문명-프리셋) 참고) |
| 4 | **이벤트 한글명·신규/기존** | "한글 이름은 '[한글명]', 구분은 [신규/기존]으로 할까요?" |

- "같은 화면에서 더 정의할 **이벤트**가 있나요?" → 있으면 위 4항목 반복. 없으면 2단계 종료.

### 3단계: 프로퍼티 수집 — 고정 질의 항목

**이 단계에서는 "이 이벤트에서 어떤 속성(프로퍼티)을 함께 보낼지"만 수집한다. 이벤트 추가/변경은 묻지 않는다.**

아래 **순서를 이벤트별로** 동일하게 유지한다.

| 순서 | 수집 항목 | 질문/확인 문구 예시 |
|------|-----------|----------------------|
| 1 | **측정 목적(KPI)** | "이 **[이벤트 한글명]**을 찍어서 **무엇을 알고 싶나요?** 어떤 숫자나 비율을 보려는지 한 줄로 알려 주세요." |
| 2 | **필요 프로퍼티 도출** | "(그 목적에 맞춰) 이 이벤트에 넣을 **속성(프로퍼티)**은 [제안 목록]입니다. 더 넣거나 빼고 싶은 게 있나요?" |
| 3 | **value(샘플) 확인** | "프로퍼티 **[속성명]**의 샘플 값(value)은 무엇으로 할까요? (예: …) 없으면 제가 예시로 채울게요." → 사용자 입력 있으면 반영, 없으면 자동 생성 |
| 4 | **정리** | "한글명·영문명(camelCase)·value·설명·구분(신규/기존/변경) 이렇게 정리했어요. 수정할까요?" |

- **명시적 구분**: "지금은 **이벤트가 아니라, 이 이벤트에 붙일 속성(프로퍼티)**을 정하는 단계예요." 라고 한 번 안내해도 좋다.

### 이벤트 영문명 프리셋

동일한 행동에 대해 스킬 호출마다 단어가 달라지지 않도록 **아래 축약·표현을 고정**해서 사용한다.

| 의미 | 사용할 표기 | 사용하지 말 것 |
|------|-------------|----------------|
| 버튼 | `btn` | button, Button, 버튼 |
| 팝업 | `popup` | pop, modal, window |
| 닫기 | `close` | dismiss, exit, 닫기 |
| 열기/오픈 | `open` | show, open |
| 클릭 | `click` | tap, select, hit |
| 탭 | `tab` | 탭 |
| 보기/노출 | `view` | show, display, load |
| 검색 | `search` | query, find |
| 결과 | `result` | results, list |
| 전송/제출 | `submit` | send, post |
| 인쇄 | `print` | 인쇄 |
| 다운로드 | `download` | down, dl |
| 로드/적재 | `load` | load |

- **형식**: 항상 **snake_case**, 소문자만. 예: `click_cert_btn`, `cert_popup_close`, `search_result`, `view_detail_tab`.
- **패턴**: `[동작]_[대상]` 또는 `[대상]_[동작]`. 예: `click_ins_btn`, `popup_close`, `tab_switch`.

## 데이터 마크다운 형식

**단계 분리**: 스킬은 1→2→3→4 순서로만 진행하며, **한 번에 한 단계만** 질문한다. 해당 단계 사용자 응답·확인 후 다음 단계로 넘어간다.

스크립트(`generate_pptx.py`, `generate_xlsx.py`)가 읽는 단일 데이터 소스 형식. JSON은 사용하지 않고 **마크다운만** 사용한다.  
**의존성**(python-pptx, openpyxl)은 사용자가 미리 설치하지 않으며, **스킬 실행 시** `run_generate.py`가 필요하면 자동 설치한다.

```markdown
# [서비스명] 태깅 정의서 데이터

## 메타정보
- 서비스명: 한라투어
- 팀: A팀
- 작성자: 하니
- 작성일: 2025.03.05

## 태깅 대상 화면
- 화면1
- 화면2

## 이벤트

### 이벤트 1
- 호출 시점: TAB 클릭 시
- Event Name (영문): smn_fit_tab
- Event Name (한글): 해외 자유여행 서브메인_탭
- 캡처 경로: docs/screens/서브메인_탭.png

| Event property | property 구분 | value | description |
|----------------|---------------|-------|-------------|
| fitTab | 신규 | 항공+호텔 | 탭명 |

### 이벤트 2
- 호출 시점: 검색 버튼 클릭 시
- Event Name (영문): search_result
- Event Name (한글): 해외 자유여행 검색
- 캡처 경로: docs/screens/검색화면.png

| Event property | property 구분 | value | description |
|----------------|---------------|-------|-------------|
| keyword | 신규 | ?keyword=프라하 | 검색어 |
```

- **메타정보**: 서비스명, 팀, 작성자, 작성일은 `## 메타정보` 아래 `- 키: 값` 목록.
- **태깅 대상 화면**: `## 태깅 대상 화면` 아래 `- 항목` 목록. 없으면 개요 페이지/시트 비우기 또는 제거.
- **이벤트**: `## 이벤트` 아래 각 `### 이벤트 N` 블록에 호출 시점·Event Name(영문/한글) 목록 + (선택) **캡처 경로** + 마크다운 표(Event property, property 구분, value, description). **캡처 경로**는 해당 이벤트 슬라이드 왼쪽 40%에 넣을 이미지 파일 경로(데이터 .md 파일 기준 상대 경로).

---

## 템플릿 동적 반영 항목

스킬이 채우는 부분만 정리.

| 구분 | 동적 반영 내용 | 입력 방식 |
|------|----------------|------------|
| **파일명** | `[서비스명]` → 실제 서비스명 | 스킬 실행 시 입력 |
| **1페이지 (표지)** | 타이틀·소속·작성자·작성일 | 서비스명, 팀(예: A 팀), 작성자(예: 홍길동), 작성일 YYYY.MM.DD |
| **2페이지 (이력)** | 버전 0.0.1 고정, 변경일=스킬 실행일 YYYY.MM.DD, 변경내용=`[서비스명] 태깅 정의 초안`, 작성자=기본 "하니" 또는 입력 |
| **3페이지 (개요)** | 태깅 대상 화면 목록 | 입력 있음 → 목록 표시, 없음 → 페이지 제거 |
| **4페이지 (태깅 상세)** | 호출 이벤트명, 왼쪽 40% 이미지(없으면 와이어프레임), 오른쪽 60% 표(Event property, property 구분, value, description). value=샘플 데이터, description=샘플 데이터 설명, property 구분=신규/기존/변경 | 스킬 실행 시 입력받아 표 생성. 전역 7개 제외, camelCase 적용 |

**4단계 시각 자료 제공 방식**: (1) 채팅에 캡처 이미지 붙여넣기(드래그 앤 드롭 또는 Ctrl+V/Cmd+V), (2) 이미 저장된 이미지 파일의 경로를 텍스트로 제공. 없으면 와이어프레임만 확보.

**이미지–이벤트 매핑**(어떤 이미지를 어떤 태깅(이벤트) 슬라이드에 넣을지):

| 방식 | 설명 |
|------|------|
| **순서로 매핑** | 이벤트를 정의한 **순서대로** 이미지를 제공(첫 번째 이미지 → 첫 번째 이벤트, 두 번째 이미지 → 두 번째 이벤트). 에이전트가 이미지를 순서대로 프로젝트에 저장할 때 파일명에 이벤트 식별자를 넣으면 좋음(예: `event1_smn_fit_tab.png`, `event2_search_result.png`). |
| **이벤트별 캡처 경로** | 데이터 마크다운에서 **이벤트마다** `- 캡처 경로: 상대경로/파일명.png` 를 넣음. PPT 생성 스크립트가 이 경로를 읽어 해당 이벤트 슬라이드 왼쪽 40%에만 배치. 채팅에 붙여넣기한 이미지는 에이전트가 먼저 저장한 뒤 이 경로를 데이터 .md에 반영하면 됨. |

## 웹 태깅 방법론 요약 (주니어 기획자용)

Amplitude Data Planning, GA4·이벤트 택소노미 참고.

- **목표부터**: 비즈니스 목표 → 핵심 지표 → 필요한 사용자 행동 순. "이 데이터로 무엇을 보고 싶은가?"를 한 문장으로 정하기.
- **이벤트**: 한 행동 = 한 이벤트. **영문명은 snake_case 통일, 가능하면 짧게**(예: `click_cert_btn`, `cert_popup_close`). 명명 규칙(팀 없을 때): `동사_대상` 또는 `대상_동사`, 사용자 관점. 비슷한 행동은 이벤트 하나 + property로 구분해 확장.
- **프로퍼티**: "이 이벤트로 무엇을 측정·판단하려는가?(KPI)"를 먼저 묻고, 그에 필요한 것만 정의. "언제, 어디서, 무엇을 대상으로" 분석 가능하게. **CTA/버튼 레이블을 일괄 프로퍼티로 넣지 않음.** 전역 7개 제외, camelCase, 이벤트당 보통 20개 이내 권장.
- **범위**: "일단 다 찍자"보다 목표·지표에 꼭 필요한 이벤트부터 정의. 정의 변경 시 이력(2페이지)에 버전·변경일·작성자 남기기.

## 이벤트·속성 정의 항목 (샘플 기준)

**이벤트 수준**: 이벤트 호출 시점/트리거, Event Category, Event Name(영문/한글), 신규/기존 구분.

**속성 수준**: Event property(속성명 영문), 속성명 한글/설명, property 구분(기존/신규/변경), value(샘플 데이터), description(샘플 데이터 설명). **value는 사용자에게 샘플 확인 후 반영, 없으면 자동 생성.**

제공 형태: PPT(화면 캡처 + 우측 속성 테이블) 또는 엑셀(동일 컬럼 표).

## KPI·인터뷰 기반 프로퍼티 정의 (예: 보험가입증서 버튼)

**상황**: "보험가입증서 버튼을 새로 추가했다. 이 버튼 추가에 대한 KPI를 측정하고 싶다."

**인터뷰 관점**:
- "이 버튼/기능으로 **무엇을 알고 싶나요?**" → 예: 얼마나 버튼을 클릭하는지, 증서 팝업에서 그냥 닫는지 vs 인쇄 버튼을 클릭하는지 비율.
- 그 답에서 **측정에 필요한 정보**만 프로퍼티로 도출:
  - 클릭 수 측정 → 이벤트 `click_ins_cert_btn` 등 + (필요 시) `source` 등 최소한의 property.
  - 팝업 내 행동 구분 → 이벤트 `cert_popup_close`, `cert_popup_print` 또는 한 이벤트 + property `action`(close/print) 등.
- **CTA 전체 레이블을 기본 프로퍼티로 넣는 것**은 KPI와 무관하면 제외. "어떤 상품의 예약인지"처럼 **의사결정에 쓰는 값**만 추가.

이 관점을 스킬 3단계에서 **질의응답**으로 진행한다.
